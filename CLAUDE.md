# CLAUDE.md — Working rules for this project

These rules are binding for every session in this repo. They override default behavior. If a rule conflicts with a direct user instruction, the user wins — but surface the conflict, don't silently pick.

## Operating rules

- **Never assume. Ask first.** For any ambiguity, blocker, or fork in direction, ask an upfront question before acting. Do not guess intent and run with it. A wrong assumption that reaches code is more expensive than a question.
- **Push back freely.** You have full license to disagree. If a design, logic, library choice, or instruction looks wrong — or just improvable — say so plainly with the reasoning. Do not perform agreement.
- **Verify, don't recall.** Before recommending architecture, logic, code, or a package, validate it against current sources (web search / official docs) rather than memory. Always confirm package compatibility against the local installed versions (see Environment) before suggesting or adding a dependency — not against the latest release. For library/package/API docs, use context7 first (version-specific official docs); fall back to web search for non-library questions (architecture, comparisons, general logic).
- **No silent/implicit fallbacks — fail fast on required config.** Anything required for the server to run or start (secrets, API keys, credentials, connection URLs/endpoints) must be validated at startup and fail loud and early if missing or invalid. Never boot with a buried default that "works" with wrong config — failing in production is far costlier to debug than failing at boot. Pure behavioral-tuning knobs with a genuinely safe default (e.g. token limits, lead-time minutes, feature flags) may keep code defaults; when in doubt whether a setting is "required," ask (rule 1).
- **Fallbacks must be explicit.** When a fallback is genuinely needed, make it visible and intentional: log it, name it, and make the trigger obvious in code. No fallback should happen where a reader can't see it.
- **No global package installs.** Strictly prohibited unless explicitly shown to be necessary and approved first. Install only into the project venv / project-local node_modules. Ask before adding any new dependency.
- **Root-cause fixes only — no patch work.** Fix the actual cause, not the symptom. No band-aids, masking, suppressing errors, or work-arounds that leave the underlying problem in place. Diagnose first, fix the real thing. If a true fix is large or risky, say so and lay out the real options (rules 1 & 2) — never quietly ship a patch in its place.
- **Log before touching anything — mandatory for every bug and every change of substance.** One tracking file, no exceptions, no branch for "too small to log":
  - **Log first.** Every bug, unexpected behavior, UI issue, feature, or refactor of substance gets an entry in `TODO.md` before any investigation starts. Bug/issue entries include a description and reproduction steps; feature/refactor entries include what's being built and why.
  - **Verify the root cause before writing a word of the spec.** Read the actual code. Do not assume, do not pattern-match from memory. For non-bug changes, verify the current behavior/architecture the change touches the same way.
  - **ADR** (docs/adr/) — only when this is genuinely an architecture/design decision (new dependency category, data-model shift, cross-cutting pattern change). Write it before the spec if needed; the spec must follow what it decides. Most bug fixes and small features will never need one.
  - **Spec** (docs/specs/) — always, no size exception. A one-line fix still gets a short spec: what's broken (or being built), the verified root cause (or design rationale), and the proposed fix. Length scales with the change; existence never does. Name the working branch (and base branch) it will land on.
  - **Wait for explicit approval.** No fix/implementation code gets written until the spec is confirmed.
  - **Implement** — on the branch named in the spec, only after everything above is done. **Branch and merge operations are human-directed: don't create, switch, merge, or delete branches, or push to `main`/`master`, on your own — do so only when the human explicitly asks. By default just commit to the already-checked-out branch.**
  - **Close the loop in the same commit** — mark the `TODO.md` entry done (link the spec and the commit), update `CHANGELOG.md` with what changed, and update `docs/FLOWS.md` whenever a change adds, removes, or changes the status of a user-facing flow (✅/⚠️/🚧/❌) — it must never drift more than one commit behind reality.
  - **Token usage, response time, or step count are never valid reasons to skip or shrink any step above.** If a task feels too small or too obvious for this process to be "worth it," that feeling is the exact signal to slow down — it's precisely when shortcuts get rationalized away. Shrink the artifact (a one-line spec, a short `TODO.md` entry), never the steps.
- **Scaffold with the framework's own generator — never hand-write what it can produce.** Before creating any new framework-specific file (component, model, controller, migration, module, route, config, test stub, or an entire project skeleton), check whether the framework/tool has a native generate/scaffold command for it — e.g. `rails g`, `php artisan make:*`, `ng generate`, `django-admin startapp`, `cargo new`, `npx create-next-app`, `vue create`, `dotnet new` — and use that instead of writing the file by hand. Generators encode current-version conventions and often perform wiring (route registration, manifest updates, paired test-file creation) that's easy to miss by hand and easy to get subtly wrong from memory. Only hand-write a file when no generator exists for that file type. **If the target file already exists, stop and ask before running the generator** — never let an overwrite prompt or a force flag resolve it silently; a customized file getting silently clobbered is exactly the kind of invisible default rule 5 already forbids. Generated output still gets reviewed against the spec/principles before being treated as done — the generator earns you the scaffold, not an exemption from rule 3.

## How debugging/tracing works, end to end

> Status: **implemented and verified** — [ADR 0004](adr/0004-temporal-langgraph-orchestration.md) / [spec 0005](specs/0005-debate-workflow.md). Confirmed against a real debate run: one shared `trace_id` across the HTTP start call, the Temporal Workflow, and every Activity; `debate_id`/`session_id`/`user_id` present on every log line.

Three separate IDs answer three different debugging questions. They get conflated under "request ID" easily, so here's what each one is actually for and where to find it.

| ID | Answers | Where it comes from | Where it shows up |
|---|---|---|---|
| **`trace_id` / `span_id`** (OpenTelemetry) | "What happened during *this one operation* — which service touched it, in what order, how long did each step take?" | Auto-generated by OTel the moment a traced operation starts (an HTTP request into Django/orchestrator, or a Temporal workflow/activity execution); propagated automatically across service boundaries via `temporalio.contrib.opentelemetry.TracingInterceptor` on the shared Temporal client — no code has to thread it through by hand. | Every span (visible on console output for now — decision 16's "local today, swappable to a real backend later, no code change"); read directly from `opentelemetry.trace.get_current_span()` and stamped onto every log line by our own logging filter (not `opentelemetry-instrumentation-logging` — its record injection turned out to require `set_logging_format=True`, which conflicts with a custom formatter). |
| **`debate_id`** | "Show me everything that happened across this debate's *entire* multi-hour lifecycle" — which spans many separate traces, since a Temporal workflow outlives any single HTTP request/trace. | The real `Debate.id` already in Postgres — no new ID minted. | Attached explicitly as a span attribute (`dialex.debate_id`) and log field at the top of every Activity and the debate-start endpoint — has to be explicit because a Temporal Workflow and its Activities don't share memory, so nothing propagates this implicitly the way `trace_id` does. |
| **`session_id` (+ `user_id`)** | "Which logged-in session/person was behind this" — independent of which debate; useful for auth-flow issues, or "what was this user doing" across multiple debates. | Minted once at login (a UUID, stored as a custom JWT claim) and carried forward through refresh-token rotation — this system uses stateless JWTs (decision 13), not server-side sessions, so there was no session concept to reuse otherwise. | Bound into logs/spans wherever a request carries a JWT — decoded directly by Django and by the orchestrator's `get_auth_context`, no extra lookup needed. |

**Worked example — "debate #482 looks stuck, what happened?"**
1. Filter the rotating log files (or, once swapped to a real backend, the trace UI) for `debate_id=482` — this surfaces every log line across Django, the orchestrator API process, and the Temporal worker for that debate's entire run, regardless of how many separate HTTP calls or Activity retries were involved.
2. Pick the specific round/step that looks wrong and read its `trace_id` — pulling that trace shows the exact causal chain for just that one Activity execution (which LLM call, how long it took, whether it retried) without the noise of the debate's other rounds.
3. If the report came from a user ("my debate seems broken"), their `session_id`/`user_id` (from their auth token) cross-references straight to which debates/cases they've touched, independent of knowing a specific `debate_id` up front.

# Changelog

## 2026-07-19 (consultant/case-submission stage, frontend)

- Built the frontend for spec 0009's consultation backend: a case-type picker, real-time chat with an AI consultant, and an approve button (`features/consultation/`, spec 0010). A real user can now go log in → "New case" → pick a case type → negotiate → approve → land on the freshly-created `/debates/:id`, entirely through the running app — the first time this whole path has been usable by an actual end user rather than curl.
- One small backend addition to support the picker: `GET /api/case-type-configs/` (Django, top-level per the already-planned API contract, not nested under `/api/cases/`) — deliberately minimal, returning just `type` since no UI needs `position_options`/`decision_options`/`research_guardrail_prompt` yet.
- Approve is disabled client-side until the consultant signals `ready_to_finalize` — the backend already enforces this (a `400` otherwise, spec 0009), but a disabled button is a better UX signal than letting a user hit that error.
- Verified in a real browser (Canary): login, the full picker → chat → approve flow with genuine OpenAI round trips (the consultant needed five back-and-forth turns before finalizing even when told plainly to — its own conversational behavior, not a bug), and the resulting debate page rendering the real, just-discussed case. **Zero app-level bugs found this pass** — worth stating plainly rather than manufacturing one: every one of the five milestones before this found something real during verification, and this is the first clean result, not a lower verification bar.
- Explicitly out of scope, unchanged: recovering an in-progress conversation after a page reload (no turn-history `GET` endpoint exists), editing a proposed payload before approving, live token streaming for chat, a "my consultations" list screen.

## 2026-07-18 (consultant/case-submission stage, backend)

- Built the backend for the consultant stage (ADR 0005/spec 0009): a new `ConsultationWorkflow` (Temporal) — the first long-running, externally-signaled workflow in this codebase, deliberately different from `DebateWorkflow`'s fire-and-forget shape since a consultation is a live dialogue, not an unattended background process. Turns are delivered via `@workflow.update` handlers (`submit_message`, `approve`), not signal+poll: the frontend/curl caller gets a normal synchronous request/response, backed by Temporal's retry/durability guarantees underneath, without needing WebSocket/Redis infrastructure.
- `CaseTypeConfig` gained `default_consultant_persona`/`default_participant_personas`/`default_judge_persona`/`default_max_rounds` — the mechanism that lets an approved consultation auto-create a real, fully-populated `Debate` with no admin involvement (the two required FK fields fail-fast at save time if left unconfigured; empty participants fails loudly at approval time instead of silently creating a broken `Debate`). `ConsultationSession.Status` gained `FAILED`, mirroring `DebateWorkflow`'s existing `mark_failed` pattern.
- No LangGraph checkpointer used anywhere (reaffirmed, not just assumed — verified against Temporal's own docs: "Temporal handles durability, so third-party checkpointers are not needed"). Conversation history is reconstructed from Postgres before every consultant turn, the same shape `fetch_arguments` already uses for debates.
- **Found and fixed four real bugs during implementation/verification, none of them hypothetical:**
  1. Two Temporal Activities (`debates.mark_failed` and the new `consultations` one) shared the exact same function name — Temporal registers Activities by name across the *entire* Worker, not per-package, so this crashed the whole worker process at startup, not just the new code. Renamed the new one to `mark_consultation_failed`.
  2. OpenAI's strict structured-output schema mode flatly rejects a free-form `dict` field (confirmed via the actual error: every property must be enumerated with `additionalProperties: false`, which an open-ended case payload can't satisfy). Fixed by having the consultant emit the payload as a JSON-encoded string, parsed back into a real dict in `graphs.py` — a schema-layer workaround, not a data-model change.
  3. Two related "retries forever instead of failing fast" gaps, same root cause as each other: the consultant LangGraph node had no bounded `retry_policy`, so bug #2 (before it was fixed) retried 30+ times with no end in sight instead of failing after a few attempts; separately, the approval activity's business-validation errors (missing/misconfigured `CaseTypeConfig`) were retryable by default, retrying 3 times before surfacing a permanent, non-transient failure. Fixed with an explicit bounded `RetryPolicy` on the node and `non_retryable=True` on the validation errors.
  4. The router's error handling for `WorkflowUpdateFailedError` needed to unwrap a different number of `.cause` levels depending on where the underlying `ApplicationError` was raised — directly in an Update handler (one level) vs. inside an Activity it calls (two levels, wrapped in an extra `ActivityError`) — verified empirically by introspecting the actual exception objects rather than assumed; getting it wrong silently returned an unhelpful generic message ("Activity task failed") instead of the real one.
- Verified end to end via curl against the real running stack: a multi-turn negotiation correctly reflecting accumulated history, approval creating a real `Case`+`Debate`+`DebateParticipant` set matching the configured defaults exactly, message-after-approval and approve-before-ready both failing cleanly (409/400, not hanging or 500ing), IDOR checks clean on both new endpoints (a non-owning user gets a clean 404), and an integration check chaining the newly-created debate straight into the existing `/api/debates/{id}/start` endpoint through to a real `Verdict` — proving the two features actually compose, not just work in isolation.
- Explicitly out of scope, unchanged: the frontend chat UI (a separate, not-yet-started follow-up spec — no real user can drive this flow yet), Redis/WebSocket streaming for either debates or consultation chat (decision 12, still not built), cancelling/abandoning a consultation session.

## 2026-07-18 (debate-thread wired to real orchestration data)

- New Django read endpoints, ownership-scoped by filtering `get_queryset()` (never fetch-then-check, the IDOR shape already caught once in spec 0005): `GET /api/cases/`, `/api/cases/{id}/`, `/api/debates/`, `/api/debates/{id}/`, `/api/debates/{id}/arguments/`.
- Resolved a real design gap: a real `Argument.position` (plain string) has no native place on the visualization's divergence↔convergence axis. Confirmed with the user: `CaseTypeConfig.position_options`' list order *is* the spectrum now (documented in its `help_text`, not a schema change) — `leaning = index / (len - 1)`, computed server-side in `ArgumentSerializer`. Case types with no fixed options (or a position outside the configured list) fall back to clustering by distinct position value in first-seen order.
- `debate-thread` now fetches real data via a new `DebatesApi` service instead of rendering hard-coded mock data — route is now `/debates/:id`; bare `/` shows a minimal empty state (no "browse my debates" list screen exists yet, a separate still-not-started flow). A "Start debate" button calls the orchestrator's existing start endpoint (spec 0005) for an `OPEN` debate, then polls automatically until the debate reaches a terminal status.
- Round-position labeling in the graph is now computed dynamically from whatever distinct round numbers are actually present, instead of a hardcoded 3-round lookup table — a latent limitation of the mock-only version that would have broken for any real debate with a different round count.
- Verified against a real browser (Canary): real case/status/strategy/round data renders correctly, real argument text and a real verdict display, ownership scoping returns a clean "not found" (not another user's data, not a crash) for both a non-owned and a nonexistent debate, the empty state works for bare `/`.
- **Found and fixed two real bugs during verification:**
  1. The orchestrator (FastAPI) had no CORS configuration at all — Django got `django-cors-headers` in the auth-wiring milestone, but the frontend now also calls the orchestrator directly (the start-debate button), and every such call failed at the browser's CORS preflight before ever reaching the service. Fixed with `CORSMiddleware`, mirroring Django's narrow-allow-list approach (a `cors_allowed_origins` setting, defaulting to the dev frontend origin).
  2. A polling-race bug: the polling-start condition excluded `status === 'OPEN'`, but `OPEN` is itself one of the active statuses — right after clicking "Start debate," if the backend hadn't transitioned off `OPEN` yet by the time of the follow-up refresh, polling never started at all, leaving the page frozen indefinitely even as the debate ran to completion server-side (caught via a 66-second zero-change DOM trace). Fixed by dropping the exclusion; re-verified with a gap-free trace showing OPEN→ARGUING→NO_CONSENSUS with arguments appearing live, zero manual reloads.
- Explicitly out of scope, unchanged: Redis/WebSocket live streaming (decision 12 — polling is a deliberate temporary stand-in), the consultant/case-submission stage, a "browse my debates" list screen.

## 2026-07-18 (debate-thread view state moves into the URL)

- `mode` (Minimal/Detail) and the selected argument node on the debate-thread page (spec 0001) are now read from URL query params on load and written back via `replaceUrl` on every change (spec 0007) — a specific view is now shareable/bookmarkable/deep-linkable. `theme` deliberately stays local component state, not a query param: it's a personal display preference, not "which view of this debate," and a shared link shouldn't impose the sharer's theme on whoever opens it.
- Verified against a real browser (Canary): defaults reflected into the URL on a bare load, clicking a node/toggling mode updates the right param without creating a new back-button-reachable history entry per click, a fresh navigation directly to a URL with query params loads straight into that state, and malformed query params fall back to sane defaults instead of crashing or rendering blank.
- Found and fixed a real bug this work exposed (a pre-existing gap from the auth-wiring milestone, not caused by this change): `login.ts`/`register.ts` navigated to `/` after auth without `replaceUrl`, leaving `/login`/`/register` as real history entries underneath. A back-press could re-enter one of them, get correctly bounced by `guestGuard`, but the already-mounted `DebateThread` route component doesn't re-run its constructor on that same-component redirect — so the address bar went bare while the page silently kept stale content. Fixed by adding `replaceUrl: true` to both auth navigations; re-verified clean.

## 2026-07-18 (frontend wired to real Django auth)

- Built real auth for the Angular app (spec 0006, governed by the already-existing ADR 0001 — no new architecture decisions needed): `Auth` service using Angular 22's new `@Service()` decorator (root-scoped by default, `inject()`-only — verified this is a genuine new Angular 22 feature, not a broken generator default, before adopting it), a functional HTTP interceptor (JWT attach + de-duplicated 401 refresh-retry), `authGuard`/`guestGuard` route guards, and login/register screens styled with the already-validated `--page`/`--ground` token system rather than a fresh design cycle.
- Added `django-cors-headers` (new backend dependency, confirmed with the user before adding) — the Angular dev server (`:4200`) and Django (`:8000`) are different origins, a gap this milestone's own spec missed until implementation.
- **Real browser verification (Canary), 3 rounds, 4 bugs found and fixed** — each fix re-verified against a fresh browser context rather than assumed correct:
  1. Angular's *built-in* XSRF interceptor silently skips cross-origin requests (verified against its source — compares request origin to page origin, no-ops on mismatch). Fixed by manually reading the `XSRF-TOKEN` cookie and attaching `X-XSRF-TOKEN` for requests to the trusted Django origin.
  2. No silent refresh on app startup — a hard reload always lost the in-memory access token even with a valid `refresh_token` cookie. Fixed with `Auth.restoreSession()`, called from the app initializer.
  3. Fixing #1 broke CORS entirely: `x-xsrf-token` is a non-simple header, triggering a preflight, and `django-cors-headers`' default `CORS_ALLOW_HEADERS` includes `x-csrftoken` (Django's *default* CSRF header name) but not `x-xsrf-token` (this project's deliberately renamed one). Fixed with an explicit `CORS_ALLOW_HEADERS` setting.
  4. Fixing #3 still left a 403 — from `CsrfViewMiddleware`'s `Origin` check, a protection entirely independent of CORS (easy to conflate). `CSRF_TRUSTED_ORIGINS` had no entry for the frontend's dev origin. Fixed by adding it.
  5. Also hardened `App.logout()` to navigate unconditionally in a `finally` — it previously only redirected after a successful server call, so bug #1 additionally meant a failed logout silently stranded the user with no error and no way back.
- Final round: all 7 verification checks passed clean — register → auto-login → logout (204) → re-login → full reload (session survives) → direct `/login` navigation while authenticated (redirected away) — zero console/CORS/CSRF errors.
- Debate-thread is unchanged, still mock-data-only — wiring it to real orchestration needs new Django read endpoints that don't exist yet, tracked as a separate follow-up.

## 2026-07-18 (Temporal + LangGraph debate workflow, real observability)

- Built the first real `DebateWorkflow` — a Temporal Workflow driving the sequential turn strategy (decision 2), with LangGraph as the per-turn reasoning layer (decision 3) via Temporal's official `temporalio[langgraph]` plugin. Each per-turn graph is a deliberately minimal single-node `StateGraph` calling OpenAI with structured output. Ends in a real `Verdict`, always produced even on `NO_CONSENSUS` (decision 8). New `orchestrator-worker` compose service (a Temporal Worker is a long-running poller, a different lifecycle from the request/response API process). New `POST /api/debates/{id}/start` endpoint is the trigger — decision 2's "FastAPI starts workflows."
- Convergence checking (decision 6) implemented for real: position stability, confidence-spread shrinkage, no-new-rebuttals, and uncited-position-change count (decision 4's dissent-preserving signal), combined into one score.
- Full OpenTelemetry observability across Django, the orchestrator API, and the orchestrator worker (decision 16, pulled into this milestone specifically to avoid retrofitting it into `activities.py` later): `trace_id`/`span_id` propagate automatically end-to-end via `temporalio.contrib.opentelemetry.TracingInterceptor` registered on the shared Temporal client; `debate_id`/`session_id`/`user_id` are attached explicitly at every Activity (a Workflow and its Activities share no memory, so nothing propagates that implicitly). Verified: one shared `trace_id` across the HTTP start call, the Workflow, and every Activity for a full debate run.
- Added a `session_id` JWT claim, minted once at login (`SessionTaggingTokenObtainPairSerializer`) and surviving refresh rotation automatically (verified against `simplejwt`'s actual rotation behavior — it mutates the same token in place rather than re-minting, so custom claims persist without extra code). This system has stateless JWTs (decision 13), not server-side sessions, so there was no session concept to reuse otherwise.
- Fixed a real security gap caught by automated review during implementation: the start endpoint had no ownership check (IDOR) — any authenticated user could start any debate by guessing its id. Now checks the debate's `Case.created_by` and 404s (not 403s) for a non-owner, matching decision 13a's existing authentication-vs-authorization principle for the WebSocket endpoint.
- Fixed a real fail-fast gap: `openai_api_key`/`temporal_address` are pydantic-settings required fields, but an empty string still satisfies `str` and passed validation silently — added `min_length=1` so a blank key now refuses to boot instead of failing deep inside an Activity later.
- Root-caused a wrong assumption rather than patching around it: `opentelemetry-instrumentation-logging` only populates `otelTraceID`/`otelSpanID` on log records when `set_logging_format=True`, which conflicts with using a custom formatter — dropped the dependency and read `trace_id`/`span_id` directly from `opentelemetry.trace.get_current_span().get_span_context()` in our own logging filter instead.
- Verified end to end against real OpenAI calls (not mocked): a full 2-round debate produced a correct citation-on-position-change (decision 4) when a participant changed its stance, real convergence-check signals, and a judge verdict with non-empty forced citations (decision 8); a separate run with a missing API key correctly landed on `Debate.status=FAILED` via the workflow's own failure path.
- Explicitly out of scope, per ADR 0004: parallel/judge-directed turn strategies, the preparation/research round, the consultant/`ConsultationSession` stage, Redis/WebSocket live streaming, and the notification system.

## 2026-07-18 (auth endpoints)

- Implemented `/api/auth/{register,login,refresh,logout}/` per decisions 13/13a — see [docs/specs/0003-auth-endpoints.md](docs/specs/0003-auth-endpoints.md). Hand-rolled cookie handling on `djangorestframework-simplejwt` (chose this over adding `dj-rest-auth` as a new dependency, given this project's learning focus and unconfirmed exact-version compatibility).

Django's CSRF cookie/header names set to match Angular's built-in `HttpClient` XSRF defaults (`XSRF-TOKEN`/`X-XSRF-TOKEN`) so the frontend won't need special configuration later.
- Smoke-tested `sqlacodegen` (decision 9) against the real containerized schema — generates clean Core `Table` objects with correct types (JSONB, Identity columns) and constraints. Confirms the tool works; the permanent CI-diffed artifact is built when FastAPI's own spec starts.
- 7 tests covering the full auth flow (register, weak-password rejection, cookie flags, CSRF enforcement, rotation, missing-cookie, post-logout blacklist) — all passing.
